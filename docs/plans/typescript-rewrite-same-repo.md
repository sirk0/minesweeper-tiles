# Same-repo variant: TypeScript app alongside the Python original

> Plan variant B. Alternative to the new-repo plan
> (`typescript-rewrite.md`, sibling file — **read it first**; its
> architecture, renderer, test-automation strategy, milestones, and PWA setup
> all apply unless overridden here). In this variant the TS app lives in
> `web/` inside `sirk0/minesweeper-tiles`, next to the Python game.

## Why choose this variant

The Python implementation is the living conformance oracle. One tree means
the static data both implementations need — the mode catalog, the difficulty
presets, and the expected board statistics — can be extracted **once** into
shared JSON that both languages read, so they cannot drift while both exist,
and porting is trivially diffable side by side.

## Shared static data: what moves to JSON (and what doesn't)

Grounded in the actual code:

- **`data/catalog.json`** ← `catalog.py`: `SURFACE_SPECS` (5 rows:
  key/label/prefix/is_3d/needs_mirror/boundary_components/tilt), the regular
  `TilingSpec` rows incl. `mode_overrides` (legacy names like `trigrid`,
  bare `torus`), Archimedean tiling keys + labels, `UNIFORM_TILINGS` /
  `DUAL_TILINGS` orderings, `GROUPS`, `SOLO_LABELS`. One wrinkle: today
  `chiral` is *computed* from geometry (`t.template().mirror is None`,
  catalog.py:97); in JSON it becomes *declared* data — add a Python test
  asserting the declaration matches the template geometry. The derivations
  (`MODE_LABELS`, `TILINGS`, `MODES_3D`, `allows()`, `mode_for()`) stay code,
  implemented identically in both languages.
- **`data/presets.json`** ← `presets.py`: the explicit `_PRESETS` builder args
  become `{mode: {difficulty: {builder: "square_board", args: [...]}}}`, and
  the `ARCH_PRESETS` table verbatim as
  `{tiling: {surface: {difficulty: [args]}}}`. Each side keeps a small
  builder-name → function dispatch map (`_ARCH_BUILDERS` equivalent).
  **Precision rule:** several args are irrational expressions
  (`1 + 1/(2+ROOT3)` presets.py:213, `ROOT3/2`, `21**0.5/4`,
  `0.5 + ROOT3/2`) — so the JSON is **generated by a script**
  (`scripts/export_data.py`), never hand-typed: Python floats and JS numbers
  are both IEEE-754 doubles and JSON round-trips full-precision `repr`
  exactly, keeping both runtimes bit-identical.
- **`data/conformance.json`** — generated by
  `scripts/export_conformance.py` from the Python boards: per
  mode × difficulty — cell count, mine count, Euler characteristic,
  boundary components, edge count, has-cell-cycle. Committed. A CI job
  regenerates and `git diff --exit-code`s it (guards Python refactors);
  **vitest consumes it as the conformance oracle** — this replaces variant
  A's "copy expected values from the Python tests by hand" and removes the
  drift risk entirely.
- **Not extracted** (algorithms, not data): the `ARCH_TILINGS` template
  builders, `_dual_template`, board builders, surface immersions, aperiodic
  substitutions.
- **Format: JSON, not YAML.** `json` is Python stdlib; Vite imports JSON
  natively with types; both round-trip IEEE doubles exactly. YAML adds a
  dependency on both sides plus number-parsing quirks. Cost: no comments in
  data files — today's inline comments in `presets.py`/`catalog.py` move to
  AGENTS.md, and `"_comment"` keys are allowed where local notes help.

## Extra milestone before M0: **P0 — data extraction (Python-only)**

1. Write `scripts/export_data.py`; generate `data/catalog.json` +
   `data/presets.json` from the current tables.
2. Refactor `catalog.py` and `presets.py` to load the JSON, keeping their
   public APIs identical (`SURFACE_SPECS`, `TILING_SPECS`, `build_board`, …).
   The existing pytest suite green is the refactor's safety net; add the
   chirality-declaration guard test.
3. Write `scripts/export_conformance.py`; commit `data/conformance.json`;
   add the CI sync job.
4. Update AGENTS.md ("add a tiling" now = one JSON preset row + a template
   builder; the tables' explanatory comments live here).

Then M0–M6 exactly as in variant A, with deltas: all TS paths under `web/`;
`catalog.ts`/`presets.ts` import `../data/*.json` instead of hand-porting
tables; the vitest oracle reads `data/conformance.json` filtered to a
`PORTED_MODES` list that grows per milestone until it equals the full catalog.

## CI / Pages (delta from variant A)

- `test.yml`: existing Python job (paths: `minesweeper/**`, `tests/**`,
  `data/**`) + new web job (`web/**`, `data/**`: typecheck, vitest, build,
  Playwright e2e/visual with software-WebGL Chromium) + data-sync job
  (re-run both exporters, `git diff --exit-code data/`).
- `.claude/hooks/session-start.sh`: additionally `npm ci` in `web/` so cloud
  sessions can run the web tests too.
- `deploy-pages.yml`: one Pages artifact composed of both apps during the
  transition — **pygbag build stays at the root** (existing URLs unchanged),
  **TS build mounts at `/next/`** (Vite `base` set accordingly via env). At
  parity (end of M5): swap — TS at root, pygbag under `/python/` or removed.
  Every milestone still ships a deployed playable TS build.

## Tradeoffs vs the new-repo plan

- **Pros:** zero drift between implementations (shared data + generated
  oracle); side-by-side diffable porting; one repo/CI; conformance data
  regenerable on demand forever.
- **Cons:** mixed Node+Python toolchain in one repo; Pages URL juggling
  during the transition; noisier history; an eventual decision about retiring
  the Python app.
- Rule of thumb: choose this variant if the Python game keeps living
  alongside the TS one for a while; choose the new repo if TS is a clean
  replacement and the Python repo becomes a frozen reference.
